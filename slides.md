# Claude Code ハンズオン
## 壊れた売上レポートツールを立て直す

### 課題シート

社内向け売上CSV集計ツールを題材に、
Claude Code の基本操作を一通り体験します

---

## このスライドについて

- 🎬 マークがついた工程は、講習でライブ実演を見た工程です
  - **見ただけで終わらせず、自分の手で再現するところから始めてください**
- 各工程の「プロンプト例」はそのまま貼り付けて使えます
  - ただし **自分の言葉に言い換えて試す** 方が学びは大きいです
- 参考ドキュメント: https://code.claude.com/docs/ja/

---

## Pythonが入っていない場合（uvを使う）

[uv](https://docs.astral.sh/uv/) を入れると、Python本体のインストールを含めて環境構築が1コマンドで済みます。

```bash
# macOS / Linux
curl -LsSf https://astral.sh/uv/install.sh | sh
```

```powershell
# Windows (PowerShell)
powershell -ExecutionPolicy ByPass -c "irm https://astral.sh/uv/install.ps1 | iex"
```

## 工程⓪ 準備

`workspace/` ディレクトリの中で Claude Code を起動します。
このシートはリポジトリ直下に置いたままにして、
Claude Code には実際の作業ファイルだけを見せます

```bash
cd csv-report-handson/workspace
claude
```

まず `README.md` のセットアップ手順どおり、
**5月分のレポートが出せること・6月分がエラーになること**を
ターミナルで確認しておきましょう

## 工程① 探索 <span class="tag">🎬</span> — コードを読ませる

**目的**: Claude Code が「コードベースを自分で調べる」エージェントであることを体感する

プロンプト例:

```
このリポジトリの構造と、レポート生成の処理の流れを説明して
```

**期待される結果**:
どのファイルも自分で開いていないのに、
loader → aggregate → cli の流れやデータの形式を説明してくれる

---

## 工程① 学びのポイント

- Claude Code は質問に答えるために、自分でファイルを検索・閲覧する
  （**エージェントループ**）
- 「どこに何があるか分からない」状態からのスタートで一番効く使い方
- 追加で「平均単価はどこでどう計算している?」のような
  具体的な質問もしてみる

参考: https://code.claude.com/docs/ja/how-claude-code-works

---

## 工程② 修正+テスト <span class="tag">🎬</span> — バグ報告文をそのまま渡す

**目的**: 「修正 → 検証」のループをエージェントに自走させる。permission 承認を体験する

プロンプト例:

```
@docs/bug_report.md にあるバグ報告を読んで、原因を調べて修正して。
修正できたら、再発防止の回帰テストを tests/ に書いて pytest で実行して
```

---

## 工程② 期待される結果

1. バグ報告を読み、`aggregate.py` のゼロ除算が原因だと特定する
2. 修正の diff が提示される → **内容を読んでから**承認する
3. テストを書き、pytest が緑になるまで自分で確認する
4. `--month 2026-06` が通るようになる

---

## 工程② 学びのポイント

- バグ報告文を要約せず **そのまま渡す** のがコツ。整形はエージェントの仕事
- diff 承認は「読んで許可する」が基本の型。ノールック承認の癖をつけない
- 「テストを書いて実行して」まで指示すると、動作確認までがエージェントの仕事になる

参考: https://code.claude.com/docs/ja/permission-modes

---

## 工程② 追加課題（静かなバグ）

バグ報告の P.S. に「返品が売上から引かれていない気がする」とあります

```
バグ報告のP.S.にある返品の件を調査して。返品（数量がマイナスの行)が
売上に正しく反映されているか、テストで確かめてから結論を出して
```

> クラッシュしないバグは「動いているから正しい」と見過ごされます。
> **テストを書くと初めて数字の間違いが見える**
> ——業務適用で一番大事な教訓です。

## 工程③ 巻き戻し — 失敗しても戻れることを体験する

**目的**: `/rewind` があるから安心して試行錯誤できる、という感覚を先に手に入れる

わざと雑な指示を出します:

```
report_tool のファイルを全部1つのファイルにまとめて
```

結果を確認したら（うまくいっていても）、`/rewind` を実行して
変更前のチェックポイントに巻き戻してください。
プロンプト入力が空の状態なら Esc を2回押しても巻き戻しメニューを
開けます（入力中のテキストがあるときはテキストのクリアになります）

**期待される結果**: ファイル構成が元に戻る。会話ごと巻き戻すか、
コードだけ巻き戻すかを選べることも確認しましょう

---

## 工程③ 学びのポイント

- 「失敗したら戻せばいい」と分かると、指示が大胆になり学習速度が上がる
- git を汚さずに試せるのがチェックポイントの利点。
  ただし万能ではないので、**大きな作業の前は git commit も併用**するのが実務の型

参考: https://code.claude.com/docs/ja/checkpointing

---

## 工程④ 記憶 <span class="tag">🎬</span> — CLAUDE.md でルールを教える

**目的**: プロジェクトの「仕事のやり方」を教えると、以後の出力すべてが変わることを体験する

まず生成させます: `/init`

生成された `CLAUDE.md` にルールを追記します:

```
CLAUDE.md に次のルールを追記して:
- レポートの金額は3桁区切りで表示する（例: 1,234,567円）
- レポートには必ず「作成日」と「データ件数」を記載する
- コードを変更したら必ず pytest を実行する
```

---

## ※Pythonが入ってない方

下記も追記するとClaude Codeが間違えにくくなります。

```
# pythonコマンド実行する時

このマシンには python3 がないので`uv run python` を使ってください

`uv run python -m report_tool.cli data/sales_2026.csv --month 2026-05`


# テスト実行時

pytestもuvを使ってください。

`uv run --with pytest python -m pytest tests/`

```
---

## 工程④ 反映を確認する

追記できたら、いったん `/clear` で会話をリセットしてから
再度レポート改修を頼み、**ルールが読まれた状態で出力が変わる**ことを確認します:

```
2026-06 のレポートを生成する処理を、CLAUDE.mdのルールに沿う形に直して
```

`/context` を実行すると、CLAUDE.md が実際にコンテキストへ
ロードされていることを目視できます

**学びのポイント**
- CLAUDE.md は「毎回プロンプトに書いていた前提」を置いておく場所
- ただし CLAUDE.md は **お願い（guidance）であって強制ではない**。
  守られないこともある——この限界は工程⑥で回収します

参考: https://code.claude.com/docs/ja/memory

---

## 工程⑤ 計画 — Plan mode で「作る前に読む」

**目的**: いきなり書かせず、計画を読んで軌道修正してから実装させる型を身につける

Shift+Tab を何度か押して **plan** 表示（Plan mode）に切り替えてから:

```
1〜6月の売上推移（月別合計）をレポートに追加したい。どう実装する?
```

**期待される結果**: コードは書かれず、実装計画が提示される。
計画を読んで、1つ注文をつけてみましょう:

```
表だけじゃなくて、前月比（%）の列も入れる計画にして
```

計画に納得したら承認して実装させ、pytest まで通してもらいます

---

## 工程⑤ 学びのポイント

- 大きめのタスクほど Plan mode。手戻りコストが一番高いのは「方向違いの実装」
- 計画への注文は実装後の修正より圧倒的に安い
- ライブデモの工程⑦でもこの型を使っていました

参考: https://code.claude.com/docs/ja/interactive-mode

---

## 工程⑥ 自動化 — hooks で「構造的に保証する」

**目的**: guidance（CLAUDE.md）と enforcement（hooks）の違いを、失敗体験込みで理解する

工程④で CLAUDE.md に「コードを変更したら必ず pytest を実行する」と書きました。
しかし何度か作業していると、**実行されないことがある**のに気づくはずです。
CLAUDE.md はあくまでお願いだからです

そこで、構造的に保証する仕組み＝hook を設定します:

```
.py ファイルを編集したら、そのファイルに python3 -m py_compile を自動実行する
PostToolUse hook をこのプロジェクトの settings.json に設定して
```

---

## 工程⑥ 学びのポイント

設定後、わざと構文エラーになる編集を頼むなどして、hook が動くことを確認しましょう。
`/hooks` コマンドで設定内容を確認できます

**期待される結果**: 編集のたびに構文チェックが自動で走り、
壊れた編集はその場で検出される

- **CLAUDE.md = guidance（守ってほしいこと）/ hooks = enforcement（必ず起きること）**
  の二層で考える
- 「絶対に守らせたいルール」を CLAUDE.md に書き続けるのは設計ミス。hook に昇格させる
- 社内展開するときは、この二層設計がそのままガバナンス設計になる

参考: https://code.claude.com/docs/ja/hooks-guide

---

## 工程⑦ 拡張 <span class="tag">🎬</span> — HTMLダッシュボードを作る

**目的**: 大きめの機能追加を「計画 → 承認 → 実装 → 検証」の型で完走する

Plan mode（工程⑤の型）で:

```
集計結果を可視化する単一HTMLのダッシュボード出力機能を追加したい。
- Chart.js（CDN読み込み）でカテゴリ別売上の棒グラフと月次推移の折れ線グラフ
- cli.py に --html オプションを追加
- 生成されたHTMLはブラウザで開くだけで見られること
```

計画を承認して実装させ、生成された HTML をブラウザで開いてください

**期待される結果**: `--html` 付きで実行すると `dashboard.html` が生成され、
ブラウザでグラフが表示される

---

## 工程⑦ 学びのポイント

- ここまでの積み上げ（テストがある・CLAUDE.md がある・hook がある）の上で
  大きな変更を頼むと、品質が保たれたまま拡張できる
- 「見た目が派手な成果物」は最後。順番が逆だと検証の習慣が身につかない

参考: https://code.claude.com/docs/ja/common-workflows

---

## 工程⑧ 卒業課題 — スキル化して自分の資産にする

**目的**: ここまでの一連の流れを1つのコマンドに固めて持ち帰る

スキル（カスタムスラッシュコマンド）の作成も Claude Code に頼めます:

```
月次売上レポートの生成を /report スキルにして。
- 引数で対象月（例: 2026-06）を受け取る
- Markdownレポートを生成し、--html 相当のダッシュボードも一緒に出す
- 実行後、生成したファイルのパスを報告する
```

`.claude/skills/` にスキルが生成されたら、実行してみましょう: `/report 2026-06`

**期待される結果**: 一言で、レポートとダッシュボードが再現される

## 発展課題（余力があれば）

1. **エッジケース処理**
   `data/sales_edge.csv` には欠損値・不正な日付・重複行が含まれています。
   `loader.py` を、不正行を警告付きでスキップできるように改修（テスト付きで）

2. **月指定の罠**
   `--month 2026-6`（ゼロ埋めなし）で実行すると何が起きますか?
   なぜそうなるのかを Claude Code に調査させ、直させてみましょう

3. **サブエージェント**
   `Explore` サブエージェントに調査を任せると、メインの会話の
   コンテキストを消費せずに済みます。「サブエージェントを使って
   このリポジトリの改善点を調査して」と頼んでみましょう
